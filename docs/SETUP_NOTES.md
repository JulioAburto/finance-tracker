# Finance Tracker Setup Notes

## Purpose

This document explains how to set up the Finance Tracker project locally and prepare it for implementation.

This project uses:

```txt
Next.js App Router
TypeScript
MUI
Supabase Postgres
Drizzle ORM
Vercel
pnpm
```

## Important Windows / PowerShell Note

In PowerShell, do not use `\` for multiline commands.

Incorrect in PowerShell:

```powershell
pnpm create next-app@latest finance-tracker \
  --ts \
  --eslint
```

Correct single-line PowerShell command:

```powershell
pnpm create next-app@latest finance-tracker --ts --eslint --app --src-dir --import-alias "@/*" --no-tailwind --use-pnpm
```

Correct multiline PowerShell command:

```powershell
pnpm create next-app@latest finance-tracker `
  --ts `
  --eslint `
  --app `
  --src-dir `
  --import-alias "@/*" `
  --no-tailwind `
  --use-pnpm
```

The backtick must be the last character in the line. Do not put spaces after it.

## Package Scripts

Add these scripts to `package.json`:

```json
{
  "scripts": {
    "db:generate": "drizzle-kit generate",
    "db:migrate": "drizzle-kit migrate",
    "db:push": "drizzle-kit push",
    "db:studio": "drizzle-kit studio",
    "db:seed": "tsx src/lib/db/seed.ts"
  }
}
```

Do not remove existing Next.js scripts.

Expected scripts should include at least:

```json
{
  "scripts": {
    "dev": "next dev",
    "build": "next build",
    "start": "next start",
    "lint": "next lint"
  }
}
```

If `next lint` is not generated by the current Next.js version, inspect the existing `package.json` and use the available lint command.

```txt

```


Use Supabase connection string from the Supabase Dashboard.

When using Supabase transaction pooler, configure the Postgres client with:


## Drizzle Config

Create:

```txt
drizzle.config.ts
```

Expected implementation:

```ts
import "dotenv/config";
import { defineConfig } from "drizzle-kit";

export default defineConfig({
  schema: "./src/lib/db/schema.ts",
  out: "./drizzle",
  dialect: "postgresql",
  dbCredentials: {
    url: process.env.DATABASE_URL!,
  },
  verbose: true,
  strict: true,
});
```

## Database Client

Create:

```txt
src/lib/db/index.ts
```

Expected implementation:

```ts
import { drizzle } from "drizzle-orm/postgres-js";
import postgres from "postgres";
import * as schema from "./schema";

if (!process.env.DATABASE_URL) {
  throw new Error("DATABASE_URL is not defined");
}

const client = postgres(process.env.DATABASE_URL, {
  prepare: false,
});

export const db = drizzle(client, { schema });
```

## MUI App Router Setup

Use MUI with the official App Router provider.

Create a provider component if needed:

```txt
src/app/providers.tsx
```

Expected pattern:

```tsx
"use client";

import { AppRouterCacheProvider } from "@mui/material-nextjs/v15-appRouter";
import { CssBaseline } from "@mui/material";
import type { ReactNode } from "react";

type ProvidersProps = {
  children: ReactNode;
};

export function Providers({ children }: ProvidersProps) {
  return (
    <AppRouterCacheProvider>
      <CssBaseline />
      {children}
    </AppRouterCacheProvider>
  );
}
```

Then wrap the app in:

```txt
src/app/layout.tsx
```

Example:

```tsx
import type { Metadata } from "next";
import { Providers } from "./providers";

export const metadata: Metadata = {
  title: "Finance Tracker",
  description: "Personal finance tracker",
};

export default function RootLayout({
  children,
}: Readonly<{
  children: React.ReactNode;
}>) {
  return (
    <html lang="es">
      <body>
        <Providers>{children}</Providers>
      </body>
    </html>
  );
}
```

## Next.js Rules

Use App Router only.

Do:

```txt
Use app/
Use page.tsx
Use layout.tsx
Use Server Components by default
Use Client Components only when required
Use Server Actions for mutations when appropriate
Use Route Handlers for backend endpoints when appropriate
```

Do not:

```txt
Do not create pages/api
Do not use getServerSideProps
Do not use getStaticProps
Do not expose secrets to the client
Do not move server logic into Client Components
```

## Implementation Order

### Step 1: Technical Foundation

* Confirm project runs.
* Install MUI.
* Install Drizzle dependencies.
* Configure MUI provider.
* Configure Drizzle.
* Create DB client.
* Create schema.
* Create seed.
* Generate migration.
* Run migration.
* Run seed.

### Step 2: Utilities

Create:

```txt
src/lib/money/convert.ts
src/lib/budget/status.ts
```

Must support:

* USD to NIO.
* NIO to USD.
* Rounding money to 2 decimals.
* Budget usage percentage.
* Budget status.
* Freeze category rule before day 20.

### Step 3: Transactions

Create:

```txt
src/app/transactions/page.tsx
src/app/transactions/new/page.tsx
src/features/transactions/actions.ts
src/features/transactions/queries.ts
src/features/transactions/schemas.ts
```

Must support:

* Create expense.
* List expenses.
* Edit expense.
* Delete expense.
* Validate data.
* Calculate `amountUsd` and `amountNio`.
* Store exchange rate.

### Step 4: Dashboard

Create:

```txt
src/app/dashboard/page.tsx
src/features/dashboard/
```

Must show:

* Monthly spending.
* Monthly budget.
* Category usage.
* Alerts.
* Latest transactions.
* Uncategorized transactions.

### Step 5: Categories and Rules

Create:

```txt
src/app/categories/page.tsx
src/app/rules/page.tsx
```

Must support:

* Create/edit/deactivate categories.
* Create/edit/deactivate merchant rules.
* Apply rules before AI.

### Step 6: Optional AI

Do not implement AI until the manual and rule-based flow works.

When approved, create:

```txt
src/app/api/classify/route.ts
src/server/ai/classifier.ts
```

AI should be fallback only.

## Common Mistakes to Avoid

* Building dashboard before the schema is stable.
* Treating credit card payment as a second expense.
* Treating payment method as category.
* Making AI mandatory.
* Hardcoding the exchange rate everywhere.
* Recalculating old transactions with a new exchange rate.
* Creating many abstractions before the first working flow.
* Adding login before the local MVP works.
* Adding OCR/imports before manual registration works.
* Overusing `use client`.

## Definition of Done

A setup task is complete when:

* Project runs locally.
* Dependencies are installed.
* MUI is configured.
* Drizzle is configured.
* Schema exists.
* Seed exists.
* DB scripts exist.
* Migration can be generated.
* No secrets are committed.
* `pnpm build` passes or failure is documented.
